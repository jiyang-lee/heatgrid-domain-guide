HeatGrid Domain Guide

0. 이 가이드의 목표

이 가이드는 도메인 지식이 전혀 없는 사람도 PreDist 기반 HeatGrid Agent를 설계할 수 있는 수준까지 끌어올리는 것을 목표로 한다.

이 문서를 모두 따라가면 아래를 설명할 수 있어야 한다.

  • 지역난방 기계실이 무엇을 하는지
  • PreDist가 어떤 데이터를 주는지
  • 어떤 센서가 어떤 부품 상태를 반영하는지
  • 대표 고장이 어떻게 센서에 나타나는지
  • 정비사는 어떤 절차로 일하는지
  • 운영자는 왜 리드타임만으로 판단하지 않는지
  • HeatGrid Agent는 어떤 정보를 바탕으로 우선순위와 재계획을 수행하는지

1. 데이터 연결 자료

먼저 읽을 문서

  • 01_PreDist_논문_정리.md
  • 02_PreDist_데이터셋_가이드.md

핵심 이해 포인트

  • PreDist는 지역난방 기계실의 실제 운영 시계열 + disturbance/maintenance/customer report 라벨을 제공한다.
  • HeatGrid는 이 데이터를 단순 예측용 CSV가 아니라 운영 의사결정의 입력 스트림으로 사용한다.
  • 중요한 것은 단일 값이 아니라 시간 흐름에 따른 패턴, 정상 대비 이탈, 이상 발생 전 조짐이다.

이 내용을 이해하면 할 수 있는 것

  • PreDist의 목적과 한계를 설명할 수 있다.
  • 데이터 컬럼을 설비/센서와 연결할 준비가 된다.

2. 지역난방 구조와 운영 이해 자료

2-1. 국내 지역난방 구조와 운영

먼저 03_국내_지역난방_구조와_운영_가이드.md를 읽는다.

핵심은 아래다.

  • 한국은 집단에너지 사업자 -> 열수송망 -> 사용자 기계실 -> 건물 난방/급탕 흐름으로 이해하면 된다.
  • 운영은 민원, 긴급신고, 정기점검, 설비 지원사업, 협력업체 출동이 함께 돌아간다.
  • 국내 자료를 보면 현재도 점검과 지원은 활발하지만 우선순위화·재계획 자동화는 약하다.

2-2. 해외 지역난방 구조와 운영

다음으로 04_해외_지역난방_구조와_운영_가이드.md를 읽는다.

핵심은 아래다.

  • 해외는 ETS(Energy Transfer Station) 또는 substation을 표준화된 설계·운영 대상으로 본다.
  • preventive maintenance, checklist, logbook, maintenance policy가 더 명시적이다.
  • HeatGrid는 국내 현장 흐름에 이 해외식 체크리스트·예방정비·기록 기반 운영을 소프트웨어로 끌어오는 방향으로 설계할 수 있다.

국내/해외 비교 표

항목국내해외HeatGrid 시사점
운영 주체사업자 + FM + 협력업체사업자 + 제조사 가이드 + 유지보수 표준역할 분리를 데이터 모델에 반영
설비 관점사용자 기계실, 민원, 안전, 지원사업ETS/substation 설계 표준, PM 중심구조와 운영을 함께 봐야 함
점검 방식자체점검, 정기점검, 민원 대응checklist, periodic PM, logbook작업지시서와 체크리스트 중요
예방정비 문화존재하지만 문서화/자동화가 약함maintenance policy가 명시적Agent는 PM 추천과 재계획에 적합
작업지시 체계사람 경험 의존 비중 큼기록·표준 절차 비중 큼설명형 작업지시서 필요

이 내용을 이해하면 할 수 있는 것

  • HeatGrid라는 이름이 왜 개별 기계실보다 네트워크 운영과 어울리는지 설명할 수 있다.
  • 국내와 해외 자료를 함께 보고 설계 용어를 고를 수 있다.

3. 정비사/정비회사 관점 자료

먼저 읽을 문서

  • 08_District_Energy_PM_Checklist_요약.md

핵심 이해 포인트

  • 정비사는 단순히 부품을 교체하는 사람이 아니라 이벤트 수신 -> 현장 점검 -> 원인 판단 -> 작업 수행 -> 기록을 담당한다.
  • 정비회사 관점에서는 무엇을 들고 가야 하는지, 어떤 체크를 먼저 해야 하는지, 작업 후 무엇을 기록해야 하는지가 중요하다.
  • 따라서 HeatGrid의 좋은 출력은 이상 점수가 아니라 점검 포인트가 정리된 작업지시다.

이 내용을 이해하면 할 수 있는 것

  • 정비사 입장에서 유용한 Agent 출력이 무엇인지 설계할 수 있다.

변수/센서물리 의미이상 시 해석Agent 출력으로의 연결
공급온도열이 사용자측으로 나가는 온도목표보다 낮으면 공급 부족 가능성난방품질 리스크
환수온도열을 전달한 뒤 돌아오는 온도과도하게 높으면 열교환 성능 저하 가능성효율 저하 또는 fouling 점검

7. 정상 패턴 vs 이상 패턴

정상 패턴을 먼저 알아야 이상을 읽을 수 있다.

  • 정상 시에는 공급온도, 환수온도, 유량, 차압이 부하 변화에 따라 일정한 상관관계를 가진다.
  • 이상 시에는 값 하나보다 관계의 깨짐이 더 중요하다.
  • 따라서 HeatGrid는 threshold만 보는 구조보다 패턴 이탈, 반응 지연, 명령-응답 불일치를 봐야 한다.

8. 운영 우선순위화와 재계획

운영자가 실제로 결정하는 것은 무엇이 가장 위험한가가 아니라 지금 무엇을 먼저 할 것인가다.

우선순위 입력

  • 리드타임
  • 영향도
  • 이상 신뢰도
  • 계절/시간대
  • 출동팀 수
  • 권역/이동 제약
  • 반복 이력

재계획이 필요한 상황

  • 팀 지연
  • 신규 고위험 이벤트 발생
  • 예상 부품 없음
  • 신뢰도 낮은 경보

즉, HeatGrid는 리드타임 회귀 모델에서 끝나지 않고 우선순위화 + 작업지시 + 재계획으로 이어져야 한다.

9. PreDist 데이터 해석

PreDist를 읽을 때의 질문

  1. 이 컬럼은 어떤 센서를 의미하는가
  2. 이 라벨은 실제 현장에서 어떤 사건을 뜻하는가
  3. 이 이상은 어떤 부품/운영 문제와 연결되는가
  4. 이 신호를 Agent가 받으면 어떤 액션으로 이어져야 하는가

핵심 매핑 표

PreDist 정보물리 의미현장 해석Agent 출력
disturbance 라벨정상 운전에서 벗어난 이벤트초기 이상 징후관찰/경보 후보
maintenance 라벨정비 개입 시점실제 작업 이력작업오더 학습 근거
customer report 라벨사용자 불편 보고이미 체감된 문제리드타임/민원 지표

10. HeatGrid Agent 설계 연결

HeatGrid에서 역할 분리는 아래처럼 가져가는 것이 가장 자연스럽다.

계층역할
데이터 계층MQTT/시계열 DB로 기계실 데이터 수집
모델 계층anomaly score, 리드타임, 위험도 계산
우선순위 계층영향도, 제약, 이력 반영
Agent 계층계획 수립, 작업지시 생성, 재계획
운영 인터페이스관제 대시보드, 코파일럿, 작업오더

한 문장 정의

HeatGrid는 지역난방 기계실 데이터를 읽고 이상 징후를 해석한 뒤, 제한된 자원 아래 무엇을 먼저 어떻게 대응할지 계획하고, 차질이 생기면 다시 재계획하는 운영 AIoT Agent다.
