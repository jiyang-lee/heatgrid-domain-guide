02. PreDist 데이터셋 가이드

한 줄 요약

PreDist는 "엑셀 파일 묶음"이 아니라 "한 기계실에서 시간 순서대로 벌어진 이야기"다. 센서 숫자가 줄줄이 적힌 표 옆에, "이때 이상이 있었다 / 이때 정비가 들어갔다 / 이때 민원이 왔다"는 사건이 같은 시간에 붙어 있다. 이 둘을 같이 읽는 법을 익히는 게 이 문서의 목표다.

이 문서에서 자주 나오는 용어
  • 시계열(time series): 시간 순서대로 죽 기록된 숫자. 예를 들어 1분마다 잰 공급온도가 쭉 늘어선 것.
  • 컬럼(column): 표의 세로 한 줄. 여기서는 보통 센서 하나에 컬럼 하나가 대응한다(공급온도 컬럼, 유량 컬럼 등).
  • 이벤트 라벨(event label): 특정 시점에 "무슨 일이 있었다"고 붙여둔 꼬리표. disturbance(이상), maintenance(정비), customer report(민원) 세 종류.
  • 기계실 단위: 데이터를 설비 하나가 아니라 "기계실 한 곳"을 통째로 보는 단위. 멀티 사이트 운영의 기본 묶음.
  • anomaly score(이상 점수): 지금 값이 평소 패턴과 얼마나 다른지를 숫자로 나타낸 것. 높다고 무조건 고장은 아니고 "더 봐야 함" 신호.

왜 이 문서가 필요한가

논문으로 "왜 이런 데이터가 필요한지"는 이해했어도, 실제 파일을 열었을 때 어느 줄을 어떻게 봐야 하는지 모르면 모델도 Agent도 만들 수 없다. 이 문서는 PreDist를 "낯선 파일 묶음"에서 "기계실 상태와 운영 사건이 연결된 이야기"로 바꿔 읽게 해준다.

PreDist 읽는 4단계

처음 파일을 열면 막막하다. 아래 순서대로 질문을 던지면 가장 쉽게 풀린다. 위에서 아래로, 큰 그림에서 세부로 좁혀가는 흐름이다.

1. 어느 기계실?

이 파일이 어느 기계실 이야기인지부터 확인한다. 데이터는 기계실 단위로 나뉜다.

2. 어떤 센서?

각 컬럼이 어떤 센서를 나타내는지 파악한다. 공급온도인지, 유량인지, 차압인지.

3. 무슨 사건?

이 시간대에 어떤 이벤트 라벨(이상·정비·민원)이 붙었는지 본다.

4. 무슨 뜻?

이 패턴이 어떤 부품 문제나 운영 문제로 번역되는지 해석한다.

무엇을 먼저 확인하나

파일을 받으면 아래 네 가지를 먼저 확인하면 구조가 빨리 잡힌다.

  • 파일이 기계실 기준으로 어떻게 나뉘는가
  • 센서 시계열이 어떤 간격(1분? 5분?)으로 기록되는가
  • 이상·정비·민원 사건이 어떤 방식으로 시간축에 붙는가
  • 하나의 이상 사건이 센서 변화와 이벤트 라벨에서 어떻게 동시에 드러나는가

데이터를 볼 때 꼭 기억할 세 가지

PreDist를 제대로 쓰려면 아래 세 가지 관점을 놓치면 안 된다.

  • 기계실 단위로 본다 — 설비 하나가 아니라 "운영 단위 한 곳"으로 본다. 같은 기계실 안의 센서들은 서로 연결돼 움직인다.
  • 한 점이 아니라 흐름으로 본다 — 어느 한 시점의 값보다, 시간이 지나며 값이 어떻게 변하는지가 더 중요하다.
  • 이벤트 라벨이 의미를 준다 — maintenance와 customer report 라벨이 "이 숫자 변화가 실제로 어떤 일이었는지"를 알려준다.

자주 하는 실수 (이렇게 읽으면 틀린다)

흔한 오해 세 가지
  • 센서값만 보고 바로 "고장"이라고 단정하는 것 → 실제로는 센서 자체가 잘못됐을 수도, 일시적 변동일 수도 있다.
  • customer report를 그냥 텍스트 꼬리표로만 보는 것 → 사실은 민원과 체감 품질, 즉 "사람이 실제로 불편했다"는 강력한 신호다.
  • maintenance 이벤트를 모델 채점용 정답표로만 보는 것 → 사실은 "실제로 사람이 개입했던 작업 이력"이라는 운영 정보다.

상황으로 이해하기

어떤 기계실에서 유량과 공급온도가 동시에 흔들리고, 조금 뒤에 maintenance 이벤트가 붙었다고 하자. 이건 "실제로 사람이 가서 손봐야 했던 이상"으로 읽을 수 있다. 만약 그 사이에 customer report까지 들어왔다면? 단순 이상이 아니라 사람이 불편을 느낄 만큼 영향이 컸던 사건으로 봐야 한다. 같은 센서 흔들림이라도 어떤 라벨이 붙느냐에 따라 무게가 완전히 달라진다.

HeatGrid에서 데이터가 흐르는 길

PreDist의 각 요소는 HeatGrid 안에서 입력 → 해석 → 출력으로 흐른다.

  • 입력: 기계실 단위 센서 시계열
  • 해석: anomaly score(이상 점수), 원인 후보, 영향도
  • 출력: 우선순위, 점검 순서, 작업지시, 재계획

한눈에 정리: 데이터 요소 → HeatGrid 연결

스스로 확인하기

  • 이 데이터가 "한 기계실의 연속된 이야기"라는 점을 설명할 수 있는가?
  • 이벤트 라벨이 왜 Agent 설계에서 중요한지 말할 수 있는가?
  • 센서값에서 작업지시까지 이어지는 흐름을 머릿속에 그릴 수 있는가?
