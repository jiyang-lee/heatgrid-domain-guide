# HeatGrid 출시 타당성 보고서

> **문서 역할**  
> 시장·정책·운영·타겟 검증 보고서 / 서비스 출시 판단용

> **검토 범위**  
> 국내 중심으로 지역난방·시설관리 시장, 정책 방향, 현재 정비 운영 방식, 고객군, 출시 요건을 정리하고, 해외 정비사·정비회사 운영 관행은 비교 기준으로 보완함

---

## 0. Executive Summary

| 항목 | 판단 |
|---|---|
| 서비스 가치 | **있음.** 단, `희귀한 대형 고장 예측`보다 `분산된 설비 운영 판단·우선순위화·재계획` 가치로 포지셔닝해야 함 |
| 국내 시장성 | **있음.** 집단에너지 공급은 국내 378만 가구, 전체 가구의 19% 수준이며, 허가 기준 75개 사업자·107개 사업장이 운영 중 |
| 정부/공공 적합성 | **우호적.** 산업부·국토부·한국지역난방공사 모두 디지털 전환, AI, 운영 고도화, 스마트빌딩, 설비 효율 개선 방향과 정합 |
| 현업 문제 적합성 | **높음.** 현재는 자체점검·정기점검·민원 접수·현장 출동 중심이며, 누수/이상 감시는 존재하지만 조기 판단과 작업 재계획은 약함 |
| 가장 유력한 초기 타겟 | **지역난방 사업자 운영조직 + 지역난방 사용 공동주택/업무시설을 관리하는 대형 FM사** |
| 출시 권고 | **조건부 출시 권고.** MVP는 `예측 모델`보다 `이상 선별 + 우선순위 + 작업지시 + 재계획` 중심으로 가야 함 |

### 한 줄 결론

HeatGrid는 `고장이 자주 나는 설비를 맞히는 제품`으로는 약할 수 있지만, `여러 기계실과 사용자 설비를 동시에 운영하는 조직이 어떤 작업을 먼저 보내고 어떻게 재조정할지를 돕는 운영 AIoT Agent`로는 충분히 설득력이 있다.

## 1. 왜 지금 이 서비스가 필요한가

### 1.1 국내 시장 규모와 구조

| 지표 | 근거 | 시사점 |
|---|---|---|
| 국내 집단에너지 사업자 수 | 한국에너지공단 기준, 2023년 말 허가 기준 **75개 사업자 / 107개 사업장** | 운영 주체가 한 공기업만이 아니라 다수 사업자로 분산되어 있음 |
| 국내 공급 규모 | 산업부 자료 기준, 2023년 말 **378만 가구**, 전체 가구의 **19%** | 서비스 대상이 틈새가 아니라 제도권 에너지 인프라임 |
| 한난 운영 체계 | 한국지역난방공사는 전국 다수 지사와 **24시간 긴급신고 체계**를 운영 | 중앙 운영 + 지역 출동 구조에 맞는 소프트웨어 수요가 존재함 |

### 1.2 정책과 산업 방향

| 주체 | 확인된 방향 | HeatGrid와의 연결 |
|---|---|---|
| 한국지역난방공사 | 2025년 AX 로드맵에서 `설비 진단`, `열수요 예측`, 이후 `AI Agent 기반 완전한 지능화·능동화`를 단계 목표로 제시 | HeatGrid 방향과 직접 정합 |
| 산업통상자원부 / 집단에너지 업계 | 안정적 열공급, 긴급 상황 대응, 사업자 간 자재 지원체계 강조 | 단순 예측보다 운영 대응 자동화가 중요 |
| 국토교통부 | 2025년 스마트빌딩 핵심기술 개발을 본격화하고 운영·관리 기술 고도화 방향 제시 | 건물/설비 운영 AI 서비스에 우호적 |
| 한국에너지공단 / 한난 지원사업 | PDCV 보수, 자동제어 절약모드, 보온재 교체 등 사용자 기계실 효율 개선을 직접 지원 | 정부·공공이 이미 유지보수·효율 개선에 돈을 쓰고 있음 |

### 1.3 결론

정부가 지금 당장 `지역난방 AI Agent`라는 이름의 제품을 사주겠다는 단계는 아니지만,  
`설비 진단`, `운영 디지털화`, `AI 기반 능동화`, `효율 개선`, `안전관리` 방향은 분명히 열려 있다.

즉, 이 서비스는 `정책 역행형`이 아니라 **정책 정합형**이다.

## 2. 현업은 지금 어떻게 일하는가

## 2.1 국내 지역난방/시설 운영의 현재 방식

| 현재 방식 | 확인 근거 | 한계 |
|---|---|---|
| 사업자 자체점검 + 한국에너지공단 정기점검 | 서울연구원은 집단에너지사업법에 따라 사업자 자체점검과 한국에너지공단 정기점검이 수행된다고 정리 | 점검은 존재하지만, 실시간 운영 의사결정 자동화와는 별개 |
| 고객 민원/긴급신고 기반 대응 | 한국지역난방공사는 `긴급신고 24시간 운영` | 민원 기반이면 이미 품질 저하 이후 대응하는 경우가 많음 |
| 열배관 감시 시스템 + 열화상 카메라 활용 | 서울연구원은 현재 감시 시스템과 열화상 탐사를 사용한다고 정리 | 누수 여부/위치 판단 한계, 지표면에서 이미 감지되면 늦을 수 있음 |
| 부품/설비 단위 유지보수 지원 | 한난·에너지공단은 PDCV 보수, 보온재 교체, 자동제어 절약모드 설치를 지원 | 현재도 문제는 `설비 고도화/정비`인데, 판단과 우선순위는 여전히 사람 중심 |
| 민간정비업체 출동 | 한난 공개 업체 목록과 공공데이터포털에는 열교환기 세관, 펌프/팬 교체, 밸브 보수, 배관누수 보수 같은 전문 업체 정보가 공개 | 현장은 이미 외주/협력업체가 수행하므로, 출동 판단과 작업지시 체계가 중요 |

### 2.2 국내에서 보이는 핵심 문제

1. `운영 정보는 많지만 행동 우선순위는 약하다`  
센서값, 사용량, 민원, 정기점검, 지원사업은 있으나 `오늘 무엇을 먼저 보낼지`는 사람 판단 비중이 크다.

2. `감시는 있지만 조기 판단과 재계획은 약하다`  
누수 탐사나 점검은 하지만, 다수 설비를 묶어서 `긴급도·영향도·가용 인력` 기준으로 자동 조정하는 체계는 약하다.

3. `민간정비 생태계가 이미 존재하므로 소프트웨어 접점이 분명하다`  
현장 수리 자체를 대체하는 게 아니라, `누구에게 어떤 작업을 어떤 순서로 내릴지`를 잘 결정해주는 계층이 필요하다.

## 2.3 해외 정비사/정비회사 운영 방식

해외는 `정비사의 업무 표준화`가 국내보다 더 문서화되어 있다.

| 항목 | 근거 | 시사점 |
|---|---|---|
| 정비사 기본 업무 | 미국 BLS/O*NET은 HVAC 정비사가 `점검`, `시험`, `부품 교체`, `고객과 고장 논의`, `작업기록`을 수행한다고 설명 | 현장 정비는 단순 수리가 아니라 진단·기록·커뮤니케이션까지 포함 |
| 유지보수 표준 | ASHRAE Standard 180은 상업용 HVAC의 최소 점검·유지관리 기준을 제시 | 해외는 예방점검 기준과 로그 체계가 제도화되어 있음 |
| 유지관리 정책 수립 | CIBSE Guide M은 maintenance policy와 building log book을 강조 | 운영사는 설비별 정책과 기록 체계를 먼저 만든 뒤 정비함 |
| ETS/서브스테이션 PM | IEA DHC Connection Handbook은 제조사 권고에 따른 preventive maintenance schedule을 개발하라고 명시 | district heating 계통도 결국 PM/점검 스케줄 기반 운영이 기본 |
| 서브스테이션 안전정비 | Gebwell 매뉴얼은 열교환기·제어밸브 등 유지보수 시 배수와 고온·고압 안전 절차를 명시 | 현장 작업은 안전·절차·작업순서가 핵심이며, Agent가 지시서로 연결할 수 있음 |
| 체크리스트 기반 정비 | District Energy St. Paul은 예방정비 체크리스트를 공개 | 해외 정비는 `체크리스트·작업오더·기록` 기반으로 움직이는 경향이 강함 |

### 2.4 정비사 관점에서 지금 일하는 방식 요약

- 이벤트를 받는다
- 현장 가서 계측하고 원인을 확인한다
- 필요 부품/교체 항목을 판단한다
- 작업 후 기록을 남긴다
- 다시 재발하는지 본다

즉, 현직 정비사의 일은 이미 `관측 → 판단 → 조치 → 기록` 구조다.  
HeatGrid는 이 앞단에서 `무엇을 먼저 보낼지`, `무엇을 들고 가야 할지`, `계획이 틀어지면 어떻게 바꿀지`를 더 잘 돕는 제품이어야 한다.

## 3. 이 서비스의 효용성과 가치

## 3.1 가치가 생기는 지점

HeatGrid의 가치는 `희귀한 대형 사고를 기적처럼 맞히는 것`이 아니다.

가치는 아래 네 군데에서 나온다.

| 가치 포인트 | 설명 |
|---|---|
| 이상 선별 | 많은 이벤트 중 지금 볼 것과 나중에 볼 것을 구분 |
| 우선순위화 | 리드타임만이 아니라 영향도, 민원 가능성, 계절성, 신뢰도, 권역 제약을 함께 반영 |
| 작업지시 | 출동 전 점검 포인트, 예상 원인, 준비 부품, 안전 유의사항 제시 |
| 재계획 | 팀 지연, 신규 고위험 설비 발생, 오탐 가능성 발생 시 기존 계획 조정 |

## 3.2 사업적 효용

| 이해관계자 | 기대효과 |
|---|---|
| 지역난방 사업자 | 공급 안정성 향상, 긴급 출동 감소, 민원 대응 속도 개선 |
| FM사 / 공동주택 운영사 | 사용자 설비 문제를 더 빠르게 분류하고 외주 작업을 더 효율적으로 관리 |
| 민간 정비업체 | 무의미한 출동 감소, 사전 정보가 있는 작업지시 확보, 재방문 감소 |
| 공공/정책 측면 | 에너지 효율, 설비 수명, 안전관리, 취약계층 난방 품질 개선과 정합 |

## 3.3 약한 서사와 강한 서사

| 서사 | 평가 |
|---|---|
| `기계실 고장이 매우 자주 나므로 꼭 필요하다` | **약함** |
| `분산된 설비를 적은 인력으로 운영해야 하므로 우선순위화와 재계획이 필요하다` | **강함** |
| `희귀 고장을 AI가 다 맞힌다` | **약함** |
| `부분 고장·성능 저하·제어 이상을 조기 감지하고 행동으로 연결한다` | **강함** |

## 4. 누구를 타겟으로 해야 하는가

## 4.1 우선 타겟 고객군

### A. 1차 타겟: 지역난방 사업자 운영조직

| 세그먼트 | 예시 | 구매 논리 |
|---|---|---|
| 공공 대형 사업자 | 한국지역난방공사, 서울에너지공사 | 지사/권역 단위 다수 설비 운영, AX·안전·효율 정책 정합 |
| 민간 지역에너지 사업자 | GS파워, 나래에너지서비스 등 | 열공급 안정성과 민원/장애 최소화, 열수송·사용자 설비 운영 고도화 |

**가장 강한 타겟**은 여기다.  
이들은 이미 운영 데이터, 정비 체계, 긴급 대응 구조, 자회사/협력업체 네트워크를 갖고 있어 Agent의 효과가 보이기 쉽다.

### B. 2차 타겟: 지역난방 사용 건물을 관리하는 대형 FM사

| 세그먼트 | 예시 | 구매 논리 |
|---|---|---|
| 공동주택/오피스/복합시설 FM | HDC랩스, SK쉴더스 FM, KT estate 등 | 사용자 기계실·공용부 설비 운영, VOC 대응, 외주 작업관리 효율화 |

FM사는 열을 생산하지는 않지만 `사용자 기계실`, `공용부 설비`, `민원`, `외주 작업관리`를 실제로 붙잡고 있다.  
특히 공동주택·대형 오피스에서 지역난방을 쓰는 경우 HeatGrid의 가치가 직접적이다.

### C. 3차 타겟: 민간 정비업체 / 유지보수 협력사

| 세그먼트 | 예시 | 구매 논리 |
|---|---|---|
| 지역난방 전문 정비업체 | 열교환기 세관, 펌프/밸브 교체, 배관 누수 보수 수행 업체들 | 출동 전 정보 확보, 반복 고장 이력 활용, 작업오더 표준화 |

다만 이들은 `최종 구매자`보다는 `현장 사용자`에 가깝다.  
실제 초기 판매는 발주권과 운영권이 있는 사업자/FM사가 더 적합하다.

## 4.2 구매자 / 사용자 / 영향자 구분

| 역할 | 누구 | 관심사 |
|---|---|---|
| 경제적 구매자 | 운영본부장, 기술처장, FM 본부장 | 장애 감소, 비용 절감, SLA, 민원 |
| 실사용자 | 관제사, 기계실 담당자, 현장 팀장 | 오늘 뭘 먼저 봐야 하는지, 오탐 줄이기 |
| 영향자 | 안전팀, 품질팀, 디지털전환 조직, 협력업체 | 도입 리스크, 데이터 품질, 현장 수용성 |

## 5. 문제 정의와 솔루션 요건

## 5.1 고객이 실제로 겪는 문제

| 문제 | 현상 | HeatGrid가 풀어야 하는 포인트 |
|---|---|---|
| 이벤트 과다 | 민원, 점검, 센서 이상, 제어 이상이 한꺼번에 발생 | 무엇을 먼저 볼지 자동 분류 |
| 리드타임만으론 부족 | 짧은 리드타임이 항상 가장 중요한 건 아님 | 영향도·신뢰도·가용 자원까지 반영 |
| 현장 정보 단절 | 관제실과 정비사 사이에 설명 손실 발생 | 작업지시서와 이력 연결 |
| 재계획 부재 | 팀 지연, 신규 이상, 부품 부족 시 사람 머리로 다시 조정 | 동적 우선순위 재산정 |
| 기록 미활용 | 반복 고장/세관/교체 이력 누적돼도 재활용이 약함 | SOP + 이력 기반 추천 |

## 5.2 반드시 들어가야 하는 솔루션 구성

### MVP 필수

| 모듈 | 내용 |
|---|---|
| 이상 이벤트 수집 | 기계실/BMS/SCADA/계측 데이터 수집 |
| 우선순위 엔진 | 리드타임 + 영향도 + 신뢰도 + 제약 반영 |

| 로그/이력 저장 | 조치 결과, 재발 여부, 작업 완료 기록 |

| 성과지표 | 오탐률, 평균 출동시간, 미조치 고위험 설비 수, 반복 고장률 |

## 6. 출시 전략 제안

## 6.1 어떤 형태로 내야 하는가

출시 초기는 `완전 자동 출동 Agent`보다 아래 형태가 더 현실적이다.

### 권장 제품 형태

`운영 추천 시스템 + 재계획 보조 Agent`

즉,
- 처음에는 `추천`
- 운영자가 검토·승인
- 현장 결과를 다시 학습

이 구조가 현업 저항이 가장 적다.

## 6.2 초기 파일럿 추천

| 항목 | 추천 |
|---|---|
| 파일럿 고객 | 지역난방 사업자 1곳 또는 대형 FM사 1곳 |
| 자산 규모 | 20~100개 기계실/건물 수준 |
| 목표 | 민원·긴급출동 대응이 아니라 `우선순위화 정확도`와 `작업 준비도 개선` 검증 |
| 기간 | 난방 피크 전후 포함 2~4개월 |

## 6.3 영업 메시지

### 사업자용

> 여러 기계실에서 발생하는 이상 이벤트를 하나의 운영 큐로 묶고, 영향도와 가용 자원을 반영해 오늘 처리할 작업을 자동 추천하는 지역난방 운영 Agent

### FM사용

> 공동주택·오피스 지역난방 사용자 기계실의 이상 신호, 민원, 외주 작업 이력을 묶어 설비팀과 협력업체 출동 순서를 자동 정리하는 시설운영 Agent

## 7. 출시해도 되는가

## 7.1 Go / No-Go 판단

| 질문 | 답 |
|---|---|
| 효용성이 있는가 | **예** |
| 시장이 충분한가 | **예** |
| 정부/공공 방향과 맞는가 | **예** |
| 현업 워크플로우에 붙일 자리가 있는가 | **예** |
| 단순 예측 모델만으로 충분한가 | **아니오** |
| 따라서 Agent가 필요한가 | **예.** 특히 우선순위화·재계획·작업지시에서 필요 |

## 7.2 단, 이런 경우는 피해야 한다

| 비추천 상황 | 이유 |
|---|---|
| 기계실 1~2개만 운영하는 단일 소규모 현장 | HeatGrid의 `Grid` 가치가 약함 |
| 데이터 없이 민원 기록만 있는 현장 | Agent 이전에 계측·기록 체계부터 필요 |
| 관제 승인 없이 완전 자동 출동을 바로 원하는 조직 | 초기 현장 신뢰 확보가 어려움 |
| 열수송관 구조건전성만 다루는 프로젝트 | 별도 안전진단/관로 모니터링 성격이 더 강함 |

## 8. 최종 권고

### 권고 포지셔닝

HeatGrid는 아래처럼 정의하는 것이 가장 납득 가능하다.

> **HeatGrid는 지역난방 사업자와 시설운영 조직이 여러 기계실과 사용자 설비에서 발생하는 이상 이벤트를 통합 관리하고, 리드타임·영향도·가용 자원·현장 차질을 반영해 작업 우선순위와 재계획을 자동 추천하는 운영 AIoT Agent다.**

### 권고 타겟 순서

1. `한국지역난방공사 / 서울에너지공사 / GS파워 / 나래에너지서비스` 같은 지역난방 사업자
2. `HDC랩스 / SK쉴더스 FM / KT estate` 같은 대형 FM사
3. 민간 정비업체 네트워크

### 권고 MVP 범위

- 고장 예측 자체보다 `우선순위 엔진`
- 관제 설명보다 `작업지시 초안`
- 챗봇보다 `재계획 루프`

즉, 이 서비스는 `AI가 고장을 다 맞힌다`가 아니라  
`사람이 이미 하고 있는 운영 판단과 출동 조정을 더 빠르고 일관되게 만든다`는 방향으로 출시해야 한다.
