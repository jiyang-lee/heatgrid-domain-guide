# AIoT Agent 고도화 방향 정리

## 1. 핵심 방향

AIoT Agent는 단순히 센서 데이터를 입력받아 응답하는 수준을 넘어, **스스로 성능을 점검하고 개선할 수 있는 구조**를 가져야 한다.

이를 위해 Agent는 다음 두 가지 방향으로 성장해야 한다.

1. **평가 지표 기반 성능 개선**
2. **실패 원인 분석 기반 자기 수정 및 자가 학습**

즉, Agent가 현재 주어진 평가 지표에 얼마나 부합하는지 지속적으로 판단하고, 부족한 부분이 발견되면 적절한 방식으로 개선해야 한다.

---

## 2. Agent 성장 방식

Agent가 모든 문제를 모델 학습으로 해결할 필요는 없다.
문제의 성격에 따라 개선 방식을 구분해야 한다.

## 2.1 모델 학습이 필요한 경우

다음과 같은 경우에는 모델 학습 또는 재학습이 필요하다.

* 센서 데이터 기반 예측 성능이 낮은 경우
* 이상 탐지 정확도가 부족한 경우
* 고장 가능성, 위험도, 상태 분류 결과가 부정확한 경우
* 기존 데이터로는 새로운 패턴을 충분히 반영하지 못하는 경우
* 평가 지표상 모델 성능 개선이 필요한 경우

이 경우에는 데이터 추가 수집, 전처리 개선, 피처 엔지니어링, 모델 재학습, 성능 평가 과정을 통해 모델 자체의 성능을 향상시킨다.

---

## 2.2 Skill / Rule 파일로 관리해야 하는 경우

모든 오류가 모델 학습 문제는 아니다.
다음과 같은 경우에는 모델을 다시 학습시키기보다 **skill 파일 또는 rule 파일**로 관리하는 것이 적절하다.

* 특정 상황에서의 대응 절차가 명확한 경우
* 에러 발생 시 해결 순서가 정해져 있는 경우
* 센서값 해석 기준이나 임계값이 명확한 경우
* 사용자 안내 문구나 리포트 형식이 필요한 경우
* 반복적으로 발생하는 오류에 대한 해결 가이드가 필요한 경우
* Agent가 참고해야 할 운영 규칙이 필요한 경우

이 경우에는 Agent가 사용할 수 있는 해결 가이드, 판단 규칙, 대응 매뉴얼을 별도 파일로 정리하고, 필요할 때 해당 내용을 조회하도록 설계한다.

---

## 3. 평가 지표 기반 개선 구조

Agent는 사용자가 만든 평가 지표에 부합하는지 지속적으로 점검해야 한다.

평가 기준은 다음과 같은 항목으로 구성할 수 있다.

* 4주 내 구현 가능한 MVP 구조인가?
* AIoT Agent 서비스로서 가치가 있는가?
* 공개 데이터셋 또는 센서 데이터를 의미 있게 활용하는가?
* LangChain 또는 LangGraph 기반 Agent 구조가 필요한가?
* 하드웨어 목업 없이도 소프트웨어 중심으로 동작 가능한가?
* 전공 지식과 NOVA 학습 내용이 자연스럽게 연결되는가?
* PHM 또는 PdM 관점에서 상태 진단, 예측, 유지보수 의사결정을 지원하는가?

Agent는 실행 결과가 이 평가 기준에 맞는지 확인하고, 부족한 항목이 있으면 개선 방향을 제안해야 한다.

---

## 4. 실패 및 오류 발생 시 자기 수정 구조

Agent가 실패하거나 오류가 발생했을 때는 단순히 오류 메시지를 출력하는 데서 끝나면 안 된다.
오류 발생 후 다음과 같은 자기 수정 흐름을 가져야 한다.

```text
오류 발생
→ 오류 로그 저장
→ 실패 원인 분석
→ 문제 유형 분류
→ 해결 방식 선택
→ 대안 탐색
→ 해결 가이드 생성 또는 수정
→ 재실행
→ 평가 지표 기반 검증
```

이 구조를 통해 Agent는 같은 오류를 반복하지 않고, 실패 경험을 새로운 지식으로 축적할 수 있다.

---

## 5. 오류 원인 분석 기준

Agent는 오류가 발생했을 때 원인을 다음과 같이 분류할 수 있어야 한다.

| 오류 유형       | 설명                         | 개선 방식                   |
| ----------- | -------------------------- | ----------------------- |
| 데이터 오류      | 결측치, 이상치, 데이터 부족, 형식 불일치   | 전처리 규칙 수정 또는 데이터 보강     |
| 모델 오류       | 예측 성능 저하, 오분류, 과적합         | 모델 재학습 또는 피처 개선         |
| Agent 판단 오류 | 잘못된 도구 호출, 잘못된 판단 흐름       | Agent rule 또는 prompt 수정 |
| 시스템 오류      | API 실패, DB 연결 실패, 파일 경로 오류 | 해결 가이드 또는 예외 처리 추가      |
| 사용자 입력 오류   | 모호한 질문, 잘못된 입력값            | 입력 검증 및 재질문 규칙 추가       |
| 서비스 설계 오류   | 사용자 가치 부족, 기능 흐름 불명확       | 평가 지표 기반 구조 재설계         |

---

## 6. 해결 가이드 생성의 필요성

AIoT Agent에서는 오류가 발생할 때마다 해결 과정을 일회성으로 끝내지 않고, 새로운 해결 가이드로 축적해야 한다.

예를 들어 다음과 같은 방식으로 관리할 수 있다.

```text
문제 상황:
센서 데이터 입력값이 비정상적으로 튀는 현상이 발생함

원인 분석:
센서 노이즈 또는 순간적인 측정 오류 가능성 존재

해결 가이드:
1. 최근 5개 측정값의 이동평균을 계산한다.
2. 이전 평균 대비 30% 이상 차이 나는 값은 이상치 후보로 분류한다.
3. 이상치가 3회 이상 반복되면 센서 오류 가능성을 사용자에게 알린다.
4. 단발성 이상치는 보정값으로 대체한다.
```

이렇게 생성된 해결 가이드는 이후 동일하거나 유사한 문제가 발생했을 때 Agent가 다시 참고할 수 있다.

---

## 7. 자기 수정 및 자가 학습 기술의 중요성

AIoT Agent가 실제 서비스로 발전하기 위해서는 단순한 응답 생성 능력보다 **자기 수정 능력**이 중요하다.

특히 PHM / PdM 기반 서비스에서는 센서 데이터, 장비 상태, 예측 결과, 사용자 피드백이 계속 변하기 때문에 Agent가 고정된 규칙만으로 동작하면 한계가 생긴다.

따라서 Agent는 다음 능력을 갖춰야 한다.

* 실패 로그를 기록하는 능력
* 오류 원인을 분석하는 능력
* 문제 유형을 분류하는 능력
* 모델 학습이 필요한 문제와 rule 수정으로 해결할 문제를 구분하는 능력
* 새로운 해결 가이드를 생성하고 저장하는 능력
* 수정된 가이드 또는 모델을 기반으로 다시 실행하는 능력
* 평가 지표에 따라 개선 여부를 검증하는 능력

---

## 8. 최종 정리

본 AIoT Agent는 단순한 센서 모니터링 시스템이 아니라, **평가 지표 기반으로 스스로 개선되는 PHM / PdM Agent**를 목표로 한다.

Agent는 실행 과정에서 발생하는 오류와 실패를 기록하고, 원인을 분석한 뒤, 문제 유형에 따라 모델 학습, rule 수정, skill 파일 업데이트, 해결 가이드 생성 중 적절한 개선 방식을 선택한다.

이를 통해 Agent는 시간이 지날수록 더 정확한 상태 진단, 예측, 조치 추천을 수행할 수 있으며, 실제 서비스 환경에서 반복적으로 발생하는 문제에 대응할 수 있는 자기 수정형 AIoT Agent로 발전할 수 있다.

---

# 한 문장 요약

AIoT Agent는 센서 데이터를 분석하는 데서 끝나는 것이 아니라, 실패를 기록하고 원인을 분석하며, 필요한 경우 모델을 학습하거나 skill/rule 파일을 업데이트해 스스로 성능을 개선하는 구조로 설계되어야 한다.

본 프로젝트는 AIoT Agent가 센서 데이터 기반 상태 진단과 예측을 수행하는 것에서 나아가, 실행 실패 및 오류 발생 시 원인을 분석하고 해결 가이드를 생성하는 자기 수정 구조를 포함한다. Agent는 문제 유형에 따라 모델 재학습, rule 업데이트, skill 파일 수정 중 적절한 개선 방식을 선택하며, 이를 통해 평가 지표에 부합하는 방향으로 성능을 지속적으로 향상시킨다.
